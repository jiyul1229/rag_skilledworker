# Pipe TIG Welding RAG Dataset

은퇴 숙련 용접공의 **노하우**를 신입에게 전달하기 위한 챗봇용 데이터셋.
"교과서적인 TIG 정의"가 아니라 **현장에서 숙련공이 판단하는 기준**을 담는 것이 목표.

---

## 폴더 구성

```
dataset/
├── video/             원본 영상 (mp4)
├── chatbot_docs/      재질별 종합 가이드 (사람이 읽는 .md, 챗봇이 긴 컨텍스트로 통째 인용)
├── rag/               RAG 임베딩용 구조화 데이터 (.json)
│   ├── carbon_steel/  1G.json / 2G.json / 5G.json / 6G.json
│   ├── stainless/     1G.json / 2G.json / 6G.json
│   ├── aluminum/      2G.json / 5G.json
│   └── posture/       6G_posture.json   ← 6G 자세 세부 분해
└── sources.json       영상 ID ↔ 로컬 경로 매핑 (인용 근거)
```

---

## 왜 이렇게 라벨링했나

### 1. `rag/`를 **재질 → 자세(1G~6G)** 2단계로 나눈 이유

같은 자세(예: 5G)라도 **재질이 바뀌면 전류·가스·송급·결함 양상이 완전히 달라진다**. 그래서 1차 분류는 재질, 2차 분류는 자세로 잡았다. 챗봇이 "스테인 6G 백비드 검게 나와요" 같은 질문을 받으면 `stainless/6G.json`만 좁게 검색하면 된다.

### 2. `posture/`가 **6G 하나**만 있는 이유

1G(아래보기)는 자세가 거의 문제되지 않는다. 자세가 결과를 좌우하는 건 **6G(45° 고정관)** 부터다. 6G는 한 패스 안에서 중력 방향이 360° 바뀌고, 모재 높이·시선·무게중심·발 위치·호흡까지 동기화돼야 한다.

그래서 **자세 자체를 독립 카테고리로 분리**하고, 6G에 한정해서 (모재 높이 / 기본자세 / 시선·무게중심·호흡 / 낮은자세 / 높은자세) 단위로 쪼갰다. 이게 숙련공이 신입에게 실제로 가르치는 단위이기도 하다.

### 3. `chatbot_docs/`를 별도로 둔 이유

`rag/*.json`은 짧은 청크(팁 한 줄, 불량 한 케이스) 단위라 임베딩 검색에 강하지만 **맥락이 짧다**. 챗봇이 답변을 풀어 쓰려면 긴 호흡의 설명이 필요해서, 재질별로 한 편씩 마크다운 가이드를 둔다. 챗봇은 검색된 청크 + 해당 재질의 .md 일부를 함께 보고 답한다.

### 4. `sources.json`이 **로컬 영상만** 인덱싱하는 이유

이 데이터셋의 인용 근거는 **로컬에 실재하는 영상**으로 한정한다. 외부 URL이나 미수집 영상은 인덱스에서 빠진다(추적은 `sources.json`의 `missing_videos` 필드).

---

## 데이터 흐름 (한 줄 요약)

`video/` 원본 → 핵심을 뽑아 `rag/<재질>/<자세>.json`(짧은 검색용) + `chatbot_docs/<재질>.md`(긴 컨텍스트용) → 챗봇이 검색 시 `sources.json`으로 원본 영상까지 역추적해 인용.
